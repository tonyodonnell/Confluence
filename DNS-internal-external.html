How DNS Resolution Works in AWS

💡
The key to understanding AWS DNS is the Route 53 Resolver — a built-in, highly available DNS service running at the VPC base IP + 2 (e.g. 10.0.0.2 in a 10.0.0.0/16 VPC). Every DNS query from inside a VPC hits this resolver first. From there, the resolver decides whether to answer from a Private Hosted Zone, forward externally, or use Resolver Rules.

Two Distinct DNS Universes

⬡ Private Hosted Zone (PHZ)PRIVATE
A DNS namespace that is only visible inside one or more associated VPCs. Records in here resolve to private IP addresses (RFC 1918). The outside internet cannot query or see these records.
Associated with: 1 or more VPCs
Typical domain: app.internal, api.corp.local
Record targets: Private IPs, ALBs, NLBs
Split-horizon: Same name → different IP than public
⬡ Public Hosted Zone (PuHZ)PUBLIC
A DNS namespace registered on the global public internet. Any resolver worldwide can query it. Records typically point to public IPs, CloudFront, Load Balancers, or API Gateways.
Visible: Globally on the internet
Typical domain: myapp.com, api.myapp.com
Record targets: Public IPs, ELBs, CloudFront
Split-horizon: Public IP returned for same name

Route 53 Resolver — Priority Decision Order

⚠️
The VPC Resolver applies rules in a strict priority order. Resolver Rules always win over Hosted Zones when there's a match on the same domain. Understanding this prevents split-horizon surprises.
Source IP is within an associated VPC AND query matches a Private Hosted Zone domain
✅ Answer from Private Hosted Zone — return private IP record. Most specific match wins.
A Resolver Rule (FORWARD type) matches the query domain
➡️ Forward to the specified DNS endpoint IP(s) (on-prem or custom DNS server). Resolver rules take priority over hosted zones for exact match domains.
No Private Hosted Zone or Resolver Rule matches
🌐 Forward to Route 53 public resolver — will check Public Hosted Zone or recurse to authoritative internet DNS.

Internal User — VPC-Based Client to Private Application

🔒
This flow covers clients inside the VPC (EC2, ECS, Lambda in VPC, etc.) resolving to services using Private Hosted Zones. The entire DNS resolution happens within AWS infrastructure. DNS traffic never leaves the VPC via the internet.
🏠 Flow 1: Internal Client → Private Application via PHZ api.app.internal → ALB private IP
1
Application / OS DNS Stub Resolver
DNS Query ↓
The application on EC2/ECS/Lambda calls getaddrinfo("api.app.internal") or equivalent. The OS stub resolver reads /etc/resolv.conf — which AWS pre-populates via DHCP to point to the VPC DNS Resolver.
DNS Query Packet
QNAMEapi.app.internal.
QTYPEA (IPv4) or AAAA (IPv6)
QCLASSIN (Internet)
FlagsRD=1 (Recursion Desired)
Dest IPVPC_CIDR + 2 (e.g. 10.0.0.2)
Port53 UDP (falls back to TCP if >512 bytes)
2
Route 53 Resolver (VPC DNS / AmazonProvidedDNS)
Receives Query
The Route 53 Resolver (always at VPC base CIDR + 2) receives the query. It checks in this order: (1) Is there a matching Resolver Rule? (2) Is there a Private Hosted Zone associated with this VPC that has a matching domain? Searches by longest label match — api.app.internal → tries full, then app.internal, then internal.
Resolver Decision Logic
Check 1Resolver Rules (FORWARD / SYSTEM) — any match?
Check 2Associated PHZs — does any PHZ match the domain suffix?
Check 3VPC-scoped names (ec2.internal, compute.internal)
FallbackForward to Route 53 public resolver for external domains
3
Route 53 — Private Hosted Zone: app.internal
Zone Query
The resolver identifies the PHZ for app.internal is associated with the source VPC. It queries the PHZ for an A or ALIAS record matching api.app.internal. The PHZ returns the configured record — often an ALB DNS name (via ALIAS) or a direct private IP.
PHZ Record Lookup
Zoneapp.internal (Private, VPC-associated)
Record Nameapi.app.internal
Record TypeA record: 10.0.10.45 OR ALIAS → internal-alb-xxx.us-east-1.elb.amazonaws.com
TTL300s (configurable; ALIAS inherits target TTL)
4
ALIAS Resolution (if record is ALIAS type)
Nested Lookup
If the PHZ record is an ALIAS pointing to an internal ALB, Route 53 resolves the ALB's DNS name within AWS infrastructure and returns the current private IP(s) of the ALB nodes directly in the response. The client sees only an A record — the ALIAS chaining is invisible.
ALIAS Chain (Transparent)
ALIAS Targetinternal-alb-1234.us-east-1.elb.amazonaws.com
Resolved IPs10.0.10.45, 10.0.20.67 (ALB nodes in each AZ)
Returned to clientA record: 10.0.10.45 / 10.0.20.67 (multiple)
5
DNS Response → Application
Answer ↑
The Resolver returns the DNS response to the OS stub resolver, which caches it per TTL and returns the IP to the application. The application then opens a TCP connection to the returned private IP on the application port. The packet flows entirely within the VPC — no internet gateway involved.
🔀 Flow 2: Split-Horizon DNS (Same Domain, Different IP Inside vs Outside) api.myapp.com resolves differently inside VPC
💡
Split-horizon allows the same domain name to return a private IP when queried from inside the VPC and a public IP when queried from the internet. You create a PHZ with the same domain name as the Public Hosted Zone and associate it with the VPC.
1
VPC Client queries api.myapp.com
Same query: api.myapp.com A? sent to VPC Resolver at VPC+2.
2
VPC Resolver checks: PHZ associated with this VPC for myapp.com?
PHZ wins!
A Private Hosted Zone named myapp.com IS associated with this VPC. The resolver queries it — ignoring the public zone entirely. The VPC association creates a "shadow" over the public zone for clients in this VPC.
3
PHZ returns private record
PHZ Response (Inside VPC)
api.myapp.comA → 10.0.10.45 (internal ALB)
PuHZ Response (Outside VPC — what internet sees)
api.myapp.comA → 52.12.44.1 (public ALB / CloudFront)

External User — Internet Client to Public Application

🌐
External users have no access to Private Hosted Zones. Their queries travel through the global public DNS hierarchy, hitting Route 53's public name servers as the authoritative source for your domain.
🌍 Flow 3: External Browser → Public App via Public Hosted Zone api.myapp.com → CloudFront / Public ALB
1
Browser / External Client DNS Stub Resolver
DNS Query ↓
User types https://api.myapp.com. Browser checks local OS cache, then sends query to the OS stub resolver, which checks its own cache and forwards to the configured recursive resolver (ISP's resolver, 8.8.8.8, 1.1.1.1, etc.).
Initial Query from Client
QNAMEapi.myapp.com.
QTYPEA
DestClient's configured recursive resolver (e.g. 8.8.8.8:53)
2
Recursive Resolver (ISP / Public — e.g. Google 8.8.8.8)
Cache Check → Recurse
The recursive resolver checks its cache. On a miss, it starts the iterative resolution process: queries a Root DNS server to find the authoritative server for .com.
Root Server Query (if no cache)
QueriesRoot Servers (a-m.root-servers.net)
AsksWho is authoritative for .com?
ResponseREFERRAL → .com TLD servers (e.g. a.gtld-servers.net)
3
.com TLD Name Servers (Verisign)
NS Delegation
The recursive resolver queries the .com TLD servers asking about myapp.com. The TLD servers return an NS delegation referral pointing to Route 53's name servers (which you set when creating the Public Hosted Zone or registering the domain).
TLD Referral Response
ANSWERREFERRAL (no direct answer)
NS Recordsns-123.awsdns-45.com
NS Recordsns-456.awsdns-67.net
NS Recordsns-789.awsdns-01.org
NS Recordsns-1234.awsdns-78.co.uk
Glue RecordsA records for NS servers included
4
Route 53 Public Name Servers (Authoritative)
Authoritative Answer
The recursive resolver queries Route 53's authoritative name servers for api.myapp.com. Route 53 looks up the Public Hosted Zone for myapp.com and returns the matching record. If it's an ALIAS to CloudFront or ALB, Route 53 resolves that and returns A records directly.
Route 53 PuHZ Authoritative Response
Zonemyapp.com (Public Hosted Zone)
Recordapi.myapp.com ALIAS → dXXX.cloudfront.net
ResolvedA: 52.84.xx.xx, 52.84.yy.yy (CloudFront edge IPs)
TTL300s (or lower for ALIAS)
AA Flag1 (Authoritative Answer)
5
Recursive Resolver → Client Response
Answer ↑
The recursive resolver caches the result and returns the A record(s) to the client. The client OS caches the result (respecting TTL) and the browser opens a TCP connection (and TLS handshake for HTTPS) to the returned IP — a CloudFront edge node nearest to the user.
🎯 Flow 4: Route 53 Health Checks & Routing Policies Weighted / Geo / Failover at DNS layer
💡
Route 53 routing policies affect which record is returned by the authoritative name server at step 4 above. This is pure DNS-layer traffic steering — no proxy involved.
Routing Policy: LATENCY
Route 53 returns the record for the AWS region with lowest measured latency to the client's resolver IP.
Routing Policy: GEO
Route 53 maps resolver IP to geography, returns record for the matching geographic region.
Routing Policy: FAILOVER
Route 53 health checks target endpoint. If unhealthy, automatically returns secondary record. Failover is near real-time.
Routing Policy: WEIGHTED
Route 53 returns records in proportion to assigned weights — useful for canary deployments and blue/green.

Hybrid DNS — On-Premises to AWS and AWS to On-Premises

🔗
Hybrid DNS uses Route 53 Resolver Endpoints — ENIs deployed into your VPC subnets that act as DNS entry/exit points over Direct Connect or VPN. This allows on-prem clients to resolve Private Hosted Zone records, and VPC workloads to resolve on-prem DNS.
⬅️ Flow 5: On-Premises Client → AWS Private Hosted Zone (Inbound Endpoint) on-prem server resolves api.app.internal
1
On-Premises DNS Server (e.g. Windows DNS / BIND)
Conditional Forwarder
On the on-premises DNS server, a Conditional Forwarder is configured: for any query ending in .app.internal, forward to the Route 53 Resolver Inbound Endpoint IP addresses (deployed in your VPC subnets). The on-prem server sends the DNS query over Direct Connect / VPN to those IPs.
On-Prem Conditional Forward
Condition*.app.internal
Forward-To IPs10.0.1.200, 10.0.2.200 (Inbound Endpoint ENIs)
TransportUDP/TCP :53 over DX or VPN tunnel
2
Route 53 Resolver Inbound Endpoint
Receives from On-Prem
The Inbound Endpoint is a set of ENIs (minimum 2, one per AZ for HA) created via Route 53 Resolver. They receive the DNS query from on-premises over the private network connection. The ENI IPs are static and you configure them as targets in your on-prem conditional forwarder.
Inbound Endpoint Specs
TypeINBOUND (accepts queries from outside VPC)
ENIs2+ ENIs across 2+ AZs in designated subnets
Security GroupAllow inbound UDP/TCP 53 from on-prem CIDR
3
Route 53 Resolver → Private Hosted Zone
PHZ Lookup
The query is now inside the VPC DNS resolution path. The resolver checks associated PHZs — finds app.internal — and returns the private A record for api.app.internal.
4
Response travels back to On-Prem DNS Server → Client
The private IP is returned over the DX/VPN link back to the on-prem DNS server, which forwards it to the on-prem client. The client can now connect to the private IP (reachable via DX/VPN routing).
➡️ Flow 6: VPC Client → On-Premises DNS (Outbound Endpoint + Resolver Rule) EC2 resolves server.corp.local (on-prem hostname)
1
EC2 Instance queries server.corp.local
Query goes to VPC Resolver at VPC+2 as normal.
2
Route 53 Resolver checks Resolver Rules
FORWARD rule match
A Resolver Rule of type FORWARD is configured in Route 53: for corp.local, forward to the on-prem DNS server IPs. The Resolver Rule is associated with the VPC. The rule matches — query is forwarded.
Resolver Rule Configuration
Rule TypeFORWARD
Domaincorp.local
Target IPs192.168.10.53, 192.168.20.53 (on-prem DNS)
VPC AssociationAssociated with this VPC
3
Route 53 Resolver Outbound Endpoint
Sends to On-Prem
The Outbound Endpoint is another set of ENIs (2+ AZs). The Resolver sends the forwarded query from the Outbound Endpoint ENI IPs out over the DX/VPN connection to the on-prem DNS server IPs specified in the Resolver Rule.
Outbound Endpoint
TypeOUTBOUND (sends queries to external DNS)
Source IPOutbound ENI IP (e.g. 10.0.3.100)
Dest192.168.10.53:53 (on-prem DNS)
4
On-Premises DNS returns answer → EC2
On-prem DNS resolves server.corp.local → 192.168.5.10. Response travels back through Outbound Endpoint → VPC Resolver → EC2. EC2 connects to on-prem server over DX/VPN.

DNS Component Reference

Route 53 Resolver VPC Built-In
The invisible DNS server always present in every VPC at VPC_CIDR + 2. Handles all intra-VPC DNS resolution. Checks PHZs, Resolver Rules, and forwards to public DNS if no match.
Address: VPC_CIDR + 2 (e.g. 10.0.0.2)
Also: 169.254.169.253 (link-local)
Port: UDP/TCP 53
Managed by AWS — no config needed to use
Private Hosted Zone (PHZ) PRIVATE
A DNS zone only resolvable from associated VPCs. Supports all standard record types (A, AAAA, CNAME, MX, TXT, SRV) plus AWS-specific ALIAS records.
Scope: VPC-scoped (association required)
ALIAS: Points to AWS resources (ALB, CF, etc.)
Cross-account: Share PHZ via RAM
Split-horizon: Create PHZ with same name as public zone
Public Hosted Zone (PuHZ) PUBLIC
Authoritative DNS for a domain on the internet. Route 53 assigns 4 name servers across different TLDs for resilience. Delegate your domain registrar's NS records here.
NS servers: 4 assigned (different TLDs)
Global anycast: R53 has 100+ edge PoPs
DNSSEC: Supported for signed zones
Health checks: Integrated routing policies
Resolver Inbound Endpoint HYBRID
ENIs in VPC subnets that accept DNS queries from on-premises networks (via DX or VPN). On-prem DNS servers forward specific domains to these IP addresses.
Min: 2 ENIs across 2 AZs
Static IPs: Stable for forwarder config
Security Group: Allow UDP/TCP 53 from on-prem
Cost: Per ENI per hour + per query
Resolver Outbound Endpoint HYBRID
ENIs in VPC subnets that Route 53 Resolver uses to send forwarded DNS queries OUT to external DNS servers (on-prem, custom DNS). Used in conjunction with Resolver Rules.
Min: 2 ENIs across 2 AZs
Source IPs: Add to on-prem firewall allow list
Security Group: Allow outbound UDP/TCP 53
Paired with: FORWARD Resolver Rules
Resolver Rules ROUTING
Per-VPC routing rules for DNS queries. FORWARD rules send matching domains to specified IPs via Outbound Endpoint. SYSTEM rules override forwarding to use local resolution.
Types: FORWARD, SYSTEM, RECURSIVE
Priority: FORWARD rules > PHZ (for exact match)
Scope: Associated per VPC
Share: Via AWS RAM to other accounts
ALIAS Records AWS-SPECIFIC
AWS-specific pseudo-record type. Like CNAME but works at zone apex (naked domain). Route 53 automatically resolves the target and returns A/AAAA records, so no extra DNS round-trip for client.
Supported targets: ALB, NLB, CloudFront, S3 website, API GW, Elastic IP, Global Accelerator, another R53 record
TTL: Inherited from target
Free: No charge for ALIAS queries to AWS resources
VPC DNS Settings VPC CONFIG
Two VPC attributes control DNS behavior and must both be enabled for Private Hosted Zones and hostname resolution to work correctly.
enableDnsSupport: Must = true
(enables VPC Resolver at VPC+2)
enableDnsHostnames: Must = true
(enables EC2 public/private hostnames)
Check: AWS Console → VPC → DNS settings

DNS Resolution Decision Logic — Which Component Answers?

🧭
Use this section to determine which DNS path will be taken for a given query scenario. The decision is determined by: (1) where the client is located, (2) VPC DNS settings, (3) Resolver Rule matches, (4) PHZ associations, (5) domain type.

Client Location Decision Tree

Client Location Query Domain Resolver Used Zone Checked Record Returned
Inside VPC (EC2, ECS, Lambda) *.app.internal (matches PHZ) VPC Resolver (VPC+2) Private Hosted Zone Private IP / ALB DNS
Inside VPC api.myapp.com (PHZ exists for myapp.com) VPC Resolver (VPC+2) Private Hosted Zone (split-horizon) Private IP (internal ALB)
Inside VPC api.myapp.com (no PHZ for myapp.com) VPC Resolver → R53 Public Public Hosted Zone Public IP / CloudFront
Inside VPC *.corp.local (FORWARD Resolver Rule) VPC Resolver → Outbound Endpoint → On-Prem DNS On-Prem DNS On-prem IP
Inside VPC ec2-*.compute.internal VPC Resolver (VPC+2) VPC internal (auto) EC2 private IP
On-Premises *.app.internal (conditional forwarder set) On-Prem DNS → Inbound Endpoint → VPC Resolver Private Hosted Zone Private IP (over DX/VPN)
Internet / External api.myapp.com ISP Recursive → Root → TLD → R53 Authoritative Public Hosted Zone Public IP / CloudFront
Internet / External *.app.internal ISP Recursive → (no route to PHZ) None (NXDOMAIN or no answer) NXDOMAIN — not resolvable

Common Gotchas & Troubleshooting Decisions

PHZ not resolving from within VPC
Check: enableDnsSupport=true AND enableDnsHostnames=true on VPC. Check: PHZ is associated with the VPC. Check: Security groups/NACLs don't block UDP 53 to VPC+2.
VPC peering and PHZ resolution fails
A PHZ is not automatically shared across peered VPCs. You must explicitly associate the PHZ with the peered VPC (even cross-account). This is a common oversight.
CNAME fails at zone apex (e.g. myapp.com)
DNS spec prohibits CNAME at apex. Use an ALIAS record instead — it works at apex and supports ALB, CloudFront, API GW, etc.
On-prem cannot resolve PHZ names
Requires: (1) Resolver Inbound Endpoint deployed in VPC, (2) On-prem DNS configured with conditional forwarder pointing to Inbound Endpoint IPs, (3) Network path open (DX/VPN), (4) Security group allows UDP/TCP 53 from on-prem IP range.
Resolver Rule takes priority when NOT expected
FORWARD Resolver Rules take priority over PHZs for same-domain matches. If a Resolver Rule for myapp.com exists, it will override a PHZ for the same domain. Remove/adjust the Resolver Rule, or use SYSTEM type to bypass forwarding.
DNS latency / caching issues during deployment
Set low TTLs (60s) before changes, change the record, wait, then raise TTL back. ALIAS records inherit their target's TTL — ALB TTL is typically 60s. R53 health-check-based failover responds within 10–30s depending on health check interval.
